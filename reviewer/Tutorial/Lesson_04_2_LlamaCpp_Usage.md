# 第 4.2 課：llama-cpp-python 的實作與單例模式設計

這節課我們要深入探討 **`backend/model_engine.py`** 的第 22-85 行。這部分代碼負責將 Python 指令轉換為底層 C++ 的高性能運算。

---

## 1. 引擎的架構：類別 (Class) 與單例模式 (Singleton)

在代碼中，我們定義了 `class ModelEngine`。
雖然在具體代碼中沒有寫死 Singleton 嚴格語法，但在 `main.py` 中，我們只在啟動時實例化了一個 `model_engine = ModelEngine()`。

### 為什麼要在後端維持「唯一一個」引擎實體？
> **深度解析**：AI 模型載入到顯存 (VRAM) 是非常消耗資源的操作。如果我們每收到一個請求就建立一個新引擎，顯存會瞬間爆炸。維持一個全域唯一的實體，是保證系統不崩潰的基礎。

---

## 2. 懶加載 (Lazy Loading) 策略

請看 `load_model` 函數的開頭：
```python
def load_model(self):
    if self.llm:
        return
```
只有在真正需要模型（即第一個使用者發問）或系統啟動時，我們才執行載入。
*   如果模型已經載入過了，就直接跳過。這能防止重複載入造成的資源浪費。

## 3. `llama_cpp.Llama` 初始化參數深度拆解

這是我們與 AI 溝通的最重要入口：
```python
self.llm = Llama(
    model_path=MODEL_PATH,
    n_gpu_layers=-1, 
    n_ctx=8192,
    verbose=False
)
```

### 極深解析：
*   **`model_path`**：模型檔案的絕對路徑。我們使用了 `os.path.join` 來確保在不同電腦上都能正確找到檔案。
*   **`n_gpu_layers=-1`**：**（重點技術）** 一個 Llama 模型是由好幾十層「神經網絡層」組成的。
    - `0`: 全部在 CPU 跑（慢）。
    - `20`: 只有前 20 層在 GPU 跑（中）。
    - `-1`: 全部塞進 GPU（最快）。
*   **`n_ctx=8192`**：這是上下文視窗 (Context Window)。這決定了 AI 的「短期記憶」。8192 代表它大約能同時記住 6000 個中文字左右的對話。
*   **`verbose=False`**：我們把 `llama.cpp` 原生的一大堆除錯訊息關掉，只保留我們自定義的精美日誌。

---

## 4. 錯誤處理邏輯 (Exception Handling)

載入模型時，最常發生「找不到檔案」或「顯卡記憶體不足」的情況。
我們使用了 `try...except` 區塊：
1.  **針對性捕捉**：先檢查檔案存不存在 (`os.path.exists`)。
2.  **全面包圍**：捕捉任何未預期的錯誤，並用 `rich` 庫的 `Panel` 工具彈出紅色的警示視窗給開發者看。

---

## 🔍 教學思考：
如果我把 `n_ctx` 設定為 `32768`（更大的記憶），會有什麼後果？
> **深度解答**：您的 AI 會變得很博學強記，但 **顯存佔用會急劇飆升**。上下文視窗是需要額外的 VRAM 來存放「鍵值快取 (KV Cache)」的。對於只有 6GB 或 8GB 的一般顯卡來說，設定過大的 `n_ctx` 會導致模型載入失敗，或是推論速度大幅下降。
