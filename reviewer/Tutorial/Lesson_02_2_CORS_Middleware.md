# 第 2.2 課：Middleware 與 CORS - 搭建跨界溝通的橋樑

這節課我們要看 **`backend/main.py`** 第 59-65 行。這幾行代碼雖然短，但它是確保前端 React 能跟後端 Python 講話的關鍵。

---

## 1. 什麼是 Middleware (中間件)？

想像 Middleware 是一道 **「安檢門」**。
在一個請求 (Request) 抵達我們的「邏輯處理器」（路由）之前，它必須先經過這道門。
*   門可以檢查請求，也可以修改請求。
*   門也可以直接拒絕請求（比如沒帶通行證的人）。

---

## 2. 深度解析：CORS 問題

當您在開發時，您的 React 畫面住在 `http://localhost:5173`。
但是您的後端伺服器住在 `http://127.0.0.1:8000`。
對於瀏覽器來說，這兩者是 **「不同的家 (Origins)」**。

為了防止壞網站偷讀你的本地數據，瀏覽器有一套 **SOP (同源政策)**：預設禁止不同家的代碼互相溝通。

### 代碼拆解：
```python
app.add_middleware(
    CORSMiddleware,
    allow_origins=["*"], # 准許所有來源
    allow_credentials=True,
    allow_methods=["*"], # 准許所有動作 (GET, POST, DELETE...)
    allow_headers=["*"], # 准許所有表頭 (Content-Type 等)
)
```

### 為什麼在 Electron 裡我們要設為 `["*"]`？
在一般網站中，為了安全，我們會限制只能讓某個特定網址連進來。
但在 Electron 應用中，前端和後端都跑在同一個使用者的電腦上。為了開發方便，我們設定為廣泛接受 (`*`)。這樣您的 React 無論怎麼變換連接埠，都不會卡在安檢門外面。

---

## 3. 中間件的執行順序 (進階)

當一個請求進來時：
1.  **CORS 安檢**：檢查來源。
2.  **如果 OK**：進入下一個中間件（如果有）。
3.  **抵達路由**：例如 `/chat`。

當回應 (Response) 返回時，它會 **「倒著走」** 一次：
1.  從路由發出答案。
2.  **CORS 處理**：在答案的標頭 (Header) 蓋上「我准許跨界」的戳章。
3.  送達瀏覽器：瀏覽器看到戳章，才願意把答案秀在 Console 裡。

---

## 🔍 教學思考：
如果我把這段代碼刪掉，React 會顯示什麼？
> **深度解答**：React 會噴出一個紅色的錯誤，大意是 `Access to fetch at ... has been blocked by CORS policy`。後端其實有收到請求並處理，但瀏覽器為了保護你，會把後端傳回來的答案直接「扣留」，不讓 JavaScript 讀取。
