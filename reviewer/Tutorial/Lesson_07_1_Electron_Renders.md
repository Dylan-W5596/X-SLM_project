# 第 7.1 課：Electron 主進程 vs 渲染進程 (Main vs Renderer)

這節課我們要解析 Electron 的靈魂架構：**「雙進程架構」**。

---

## 1. 什麼是雙進程？

想像一個公司：
*   **主進程 (Main Process)**：就像是 **老板**。他有鑰匙，可以進出金庫、啟動機器、決定公司開不開門。在程式中就是 `main.cjs`。
*   **渲染進程 (Renderer Process)**：就像是 **門市人員**。他的工作是微笑、畫美美的海報（React 畫面），但他沒有鑰匙。在程式中就是我們的 `App.jsx`。

---

## 2. 為什麼要分這麼開？

**安全與穩定。**
如果門市人員（網頁）被壞人（病毒）入侵，因為他沒有鑰匙，他無法直接存取使用者的硬碟。
如果網頁當機了，老板（主進程）還活著，他可以負責關掉門市窗口並重新啟動，而不至於讓整個作業系統當掉。

---

## 3. 本專案的配置拆解

請看 `main.cjs` 第 14-25 行：
```javascript
mainWindow = new BrowserWindow({
    webPreferences: {
        nodeIntegration: true, // 准許前端讀取 Node.js 功能
        contextIsolation: false // 降低隔離強度 (開發方便用)
    }
});
```
### 極深解析：`contextIsolation`
在極安全的環境下，我們會設為 `true`。但本專案是本地運行的 AI 工具，為了讓 React 能直接透過 `ipcRenderer` 發送指令給老板，我們放寬了這個限制。這讓開發變得更直觀，不需要寫複雜的 Preload 腳本。

---

## 4. 生命週期的權限

*   **老板 (Main)**：知道 App 什麼時候啟動 (`ready`)，什麼時候要下班 (`window-all-closed`)。
*   **門市 (Renderer)**：只知道使用者有沒有按下傳送按鈕。

---

## 🔍 教學思考：
如果我把 AI 引擎寫在 `main.cjs` (老板) 裡，而不是另外跑一個 Python (外部經理)，會發生什麼？
> **深度解答**：這會導致 **「視窗卡死」**。Electron 的主進程負責處理所有的視窗 UI 事件。AI 推論是「極度耗時」的工作，如果放在主進程，當 AI 在計算時，你連拖動視窗都會卡住。這就是為什麼我們特別「外包」給 Python 的原因。
